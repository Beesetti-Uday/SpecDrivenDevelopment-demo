This repository serves as a hands-on demonstration and comparative environment contrasting unstructured AI generation with **Spec-Driven Development (SDD)**. ## 🌿 Branch Architecture
To demonstrate the procedural difference between "vibe coding" and a specification-first framework, this repository is organized into three distinct tracks:

| Branch | Development Methodology | Description |
| :--- | :--- | :--- |
| `main` | **Baseline** | The core starter application code before any features are added. |
| `w-coding` | **Vibe Coding Track** | Quick, ad-hoc feature additions generated by prompting an AI assistant directly without pre-planning documents or strict structural boundaries. |
| `openspec` | **Spec-Driven Track** | Rigorous feature tracking built using structured SDD phases (`/propose` $\rightarrow$ `/apply` $\rightarrow$ `/archive`), keeping specs and implementation tightly aligned. |

---
## 🛠️ The Spec-Driven Workflow (`openspec` branch)
The structured development path follows the open specification lifecycle to guarantee documentation stays in lockstep with the code:
1. **Constitution & Constraints:** Establishing baseline project principles and technical constraints for the AI agent.
2. **Propose Phase (`/propose`):** The AI generates localized planning files (`proposal.md` and `tasks.md`) to map out requirements *before* altering production code.
3. **Apply Phase (`/apply`):** Code changes are incrementally built, strictly bounded by the accepted design criteria and task list.
4. **Archive Phase (`/archive`):** Temporary context maps are compiled into a permanent `specs/spec.md` reference file to preserve token economy and clean up the workspace.
